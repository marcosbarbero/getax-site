Configuring GetAX

How a repository configures GetAX — and, the load-bearing part, which settings can move your score and which cannot.

GetAX's whole value is a score you can argue with: deterministic, reproducible, explainable. That only holds if the score moves only through inputs that were reviewed. So configuration splits into two files, divided by a single question.

Validate your config

Paste your .getax/settings.json below. It's checked against the engine's own rules, live, in your browser — nothing is sent anywhere. (Verified claims like exclude and archetype are checked against your repo at scan time; this validates the shape.)

The one idea: config splits by whether it can touch the score

Committed · reviewed in the diff

.getax/settings.json

You edit it. It is committed and reviewed like any code change — and it can move the score, precisely because it is reviewed.

Model pin, weights, ignore globs, report inputs, overrides, issue conventions.

Gitignored · personal

.getax/settings.local.json

You edit it, on your machine. It is gitignored and cannot touch the score of record — structurally, not by a rule anyone polices.

A local display mute. That is all it can express.

The score of record — the number your CI gates on, the ratchet's floor, anything the hosted dashboard sees — is computed from .getax/settings.json alone. The local file is never read on that path, has no field that could change a score (a weights key in it is a parse error, not a silent no-op), and is gitignored, so CI and the server never see it at all. You can silence anything locally and it cannot reach the shared number — because the shared number never looks at the local file.

Both files live under one directory, .getax/, beside .getax/baseline.json (the ratchet floor — the tool writes that one; leave it alone). Run getax fix once to scaffold both and gitignore the local one; run getax config any time to see every effective setting and which layer it came from.

.getax/settings.json — the committed, score-affecting config

A JSON object. Unknown fields are rejected — a typo'd "wieghts" is an error, not a silent no-op, because a config that says one thing while the score means another is wrong in a way you cannot see.

{
  "$schema": "https://getax.app/schema/getax.v1.json",
  "version": 1,
  "model": "v3",
  "weights": { "T":0.25, "B":0.20, "D":0.15, "S":0.20, "E":0.20 },
  "ignore": ["vendor/**", "**/*.generated.go"],
  "inputs": { "coverage": ["build/coverage.out"] },
  "overrides": [
    {
      "rule": "type-strictness",
      "reason": "Mixpanel event properties and GeoJSON — open-world JSON by definition.",
      "expires": "2027-01-01"
    }
  ],
  "issues": { "labels": ["type:task", "comp:api"] }
}
version RequiredMoves score

Must be 1. The config format version — not the scoring model (that's model). A build that doesn't understand your version refuses to score rather than guess.

model RequiredMoves score

The scoring model version to pin, e.g. "v3". This is the most important field. A CLI upgrade must never silently move your score — someone's CI ratchet depends on the number. Pinning means a new GetAX release scores you against the model you chose, not whatever shipped last night. On our side, changing a band, a weight default, or a penalty curve is a model version bump plus an ADR, never a patch — and you adopt it deliberately by changing this one line.

A repo with no .getax/settings.json is unpinned: it gets the current model, and the report says so. Unpinned is fine for a first look; pin before you gate CI.

weights OptionalMoves score

The five signal weights — T determinism, B boundaries, D intent, S type safety, E enforcement. Must sum to 1.0 — rejected, not silently rescaled, if they don't. Omit to use the model's defaults.

ignore OptionalMoves score

Path globs excluded from probing. Compiled at load, so a bad pattern fails the scan with the pattern named rather than silently matching nothing.

inputs OptionalMoves score

Where your build reports actually are, when they aren't where the conventions guess. Four kinds, each an array of path globs: coverage, junit, sarif, openapi.

This matters most for coverage: coverage reports are build artifacts (gitignored, produced wherever your build puts them), so any path GetAX hard-codes is a guess. Configured paths win over conventions — point at a path and if nothing is there, that's an error we surface, not a cue to hunt elsewhere and report a number from a file you didn't mean.

Why a report path is "moves score": finding a real report changes the number (it measures a blind spot — honestly). A machine-specific local path that shouldn't affect the shared number is a different thing and isn't yet supported in the local layer; put the canonical, CI-reachable path here.
overrides OptionalMoves score

Tell GetAX a rule is wrong for your repo, and why. This is "Override & Teach": the finding stops counting against you, deterministically, because it's a reviewed input — not a learned parameter.

FieldRequiredMeaning
ruleyesthe sub-signal key, e.g. change-traceability
reasonyeswhy it doesn't apply here — min 20 characters
expiresnoYYYY-MM-DD; the override lapses after this date
scopenoa path glob narrowing the override to part of the tree
authornowho decided (git blame answers this too)

The reason is mandatory on purpose, and it's the whole point. A reasonless override is just a lower standard with extra steps. The reasons are a teaching signal: in aggregate, "22 of 37 teams overriding this rule cite generated code" is evidence the rule is wrong, and a human fixes it in the next model version. Twenty characters doesn't guarantee thought — it guarantees skipping thought is a deliberate act.

expires exists so overrides rot loudly, not silently. An override with no expiry is a decision nobody revisits — how a repo ends up scoring well by permanent exemption. The score report lists every applied override with its reason, because determinism is "same inputs → same score, and the report names its inputs."

issues OptionalInert to score

How this repo wants findings filed: labels and assignees applied when you file a finding as a GitHub issue. Empty by default and deliberately — GitHub silently drops a label that doesn't exist in the target repo, so a guessed taxonomy files unlabelled and looks like it worked. Tell us your labels; we won't guess them.

Verified config — exclude & posture requires model v4

The principle: committed config supplies claims the engine verifies and recomputes from, never values it trusts. A raw ignore glob is trusted; a verified claim is earned. These require "model": "v4" — a repo that writes them under an older pin gets a clear error, never a silent no-op — and until you pin v4, nothing here touches your score.

exclude Optional · v4Verified

Instead of trusting a glob, you make a claim the engine checks:

"exclude": [
  { "path": "**/i18n/**", "claim": "data",
    "reason": "Localisation catalogs — key/value data, not logic.", "expires": "2027-01-01" }
]

Claims are a closed vocabulary: generated (codegen convention or a DO NOT EDIT header), vendored (directory), data (content shape — mostly key/value, almost no control flow). If the claim holds, those files are exempt from the affected signal. If it fails, nothing is exempt and the suppression ledger names the evidence: hand.go generated → REJECTED (1/1 failed). You cannot raise the score by writing a claim — only by the claim being true.

posture Optional · v4Verified

For a repo whose gate is a committed pre-push hook rather than CI:

"posture": { "gate": "pre-push-hook" }

This does not waive the enforcement signal — it redirects it. The scanner points at .githooks/pre-push and checks whether it actually runs your tests. A hook that runs them earns the same ceiling as a CI that does (a hook is per-clone, so we see the mechanism but can't confirm it's installed); a hook that runs nothing earns nothing.

archetype Optional · v4Verified

Declare what kind of repo this is, and the scorer uses a weight profile suited to it — so a CLI isn't marked down for having no HTTP API, and a data pipeline isn't marked down for long files:

"archetype": "service"
archetypeleans into
general (default)the balanced base — today's weights
servicedeterminism + enforcement (deployed: verify the change, bind the gate)
librarytype safety + intent (the API surface is the product)
clitests + gate; api-schema doesn't apply
data-heavycorrectness at the data boundary; long files are the nature, not a defect

A closed vocabulary (so scores stay comparable) and verified with evidence, as strictly as an exclude claim. The archetypes that ease a signal must earn it: library (lowest enforcement weight) is rejected on deploy markers; data-heavy (eases the boundaries weight) is rejected unless data is a real share of the repo — "data is 4% of your code+data files — this leads with code." A rejected archetype falls back to general and the report names why. The ones that only raise the hard-to-fake signals (service, cli) are accepted. You can't set both archetype and weights (an archetype is a profile).

.getax/settings.local.json — the personal, display-only layer

Gitignored, per-developer, and structurally unable to move the score of record. getax fix scaffolds it and adds it to .gitignore. Unknown fields are rejected — you cannot put a model, weights, or overrides key here; that's a parse error telling you it belongs in .getax/settings.json, where it's reviewed.

{
  "quiet": ["suite-speed", "type-strictness"]
}
quiet OptionalDisplay only

Sub-signal keys to hide from your getax scan recovery list — "I've seen this, stop reminding me on my machine." It does not change the score, which findings count, or the --json output; the finding still appears in the signal breakdown and still counts. And it's never silent: the report ledgers N quieted locally … still scored, so a local mute is visible, not invisible state.

To make a suppression travel — reviewed, applied by CI and the CLI — write an override in .getax/settings.json instead.

Seeing what's in effect: getax config

Prints every effective setting under its layer, so the firewall is visible rather than merely asserted:

  getax configuration  .

  COMMITTED  .getax/settings.json — reviewed in the diff, may move the score
    model       v3                pinned
    version     1
    weights     default
    ignore      vendor/**
    inputs      coverage:1
    overrides   1 — type-strictness
    issues      labels:type:task/comp:api

  LOCAL      .getax/settings.local.json — personal, gitignored, cannot move the score
    quiet       suite-speed       display only — these findings still count

Determinism guarantees, in one place

GuaranteeWhat it means
Same inputs → same scorebyte for byte. No wall-clock, no network, no randomness, no generative AI in scoring.
No silent score movesthe model is pinned in .getax/settings.json; adopting a new model is a line change you make.
Every point names its causeand every override that changed the number is listed with its reason. Nothing deducted unattributed.
The local layer can't gate youit's not read on the score-of-record path, and it's gitignored — CI and the server never see it.
